# 3단계 지시서 v1.1 — 콜드 해석 데모: MCP 서버 + 에이전트 플로우 (전문)

이 문서는 v1을 **대체**하는 자기완결 지시서다. 목적이 재정의됐다: 이 단계의 산출물은 "검색 서버"가 아니라 **포트폴리오의 핵심 장면** — *임의의 조서 파일을 Claude Code에 던지면, 에이전트가 읽고 해석해 할 일과 근거 규정을 뽑아내는 플로우* — 이다. MCP 서버는 그 에이전트가 드는 회계·감사 지식 도구다. 참조 계약: 규약 v2.1, `index/manifest.json`, `index/vocab.json`. 근거 있는 이탈은 README 기록 조건으로 허용.

**v1 대비 변경 지도** — ① `standards_worksheet_map` 런타임 도구 삭제(라우팅 표는 채점표로 승격: `eval/routing_gold.json`), ② 에이전트 해석 플로우 신설(6장 — CLAUDE.md 탑재), ③ 평가 절차 신설(7장 — 골드셋 recall), ④ 인수 테스트 2계층화(MCP A테스트 + 데모 D시나리오), ⑤ 데모 호스트 = Claude Code 확정(9장). 도구 3종·기동 검증·주입·정책은 v1 그대로.

**한 줄 서사 (모든 산출물이 이를 증명해야 한다):** 자유로운 해석, 검증 가능한 인용 — 에이전트는 처음 보는 조서를 자유롭게 해석하되, 인용하는 모든 규정은 복합 ID로 실물 대조가 가능하다.

---

## 0. 데모의 정의 — 합격 장면

사용자가 Claude Code에 조서 파일 하나를 주며 "이 조서 해석해줘"라고 말한다. 에이전트는 (a) 파일을 직접 읽고, (b) 무엇을 하는 조서인지 파악하고, (c) MCP 도구 3종으로 근거를 탐색해, (d) 다음 네 부분의 보고서를 산출한다: **① 조서의 정체 ② 할 일 목록(각 항목에 근거 cid) ③ 근거 규정 발췌 ④ 미확인·불확실 사항.** 인용된 cid는 전수가 `standards_get_paragraph`로 실물 검증된다. 사전 라우팅 표는 **어디에도 개입하지 않는다.**

포트폴리오는 두 축을 갖는다: 정성(위 장면의 대본·기록) + 정량(46종 골드셋 대비 콜드 해석의 근거 적중률 — 7장).

## 2. 구조와 빌드 순서

```
server/  core.py(코어: Gateway·정책·주입 — MCP 비의존) · contracts.py(기동 검증) · mcp_server.py(도구 3종)
eval/    routing_gold.json · score_interpretation.py(채점기)
reports/ 해석 보고서 저장 (해석_{조서파일명}.md)

```

빌드 순서: ① contracts + `standards_get_paragraph` 뼈대 → stdio 실연 → ② `standards_search` → ③ 정책·주입 → ④ `standards_define_terms` → ⑤ CLAUDE.md 플로우 + 파일럿 1종 콜드 해석 → ⑥ 채점기 + 파일럿 4종.

## 3. 기동 검증 (contracts.py) — v1과 동일, 전문 재수록

| 검사 | 기준 | 실패 시 |
|---|---|---|
| 임베딩 모델 | 모델명·revision·normalize·차원 == manifest.embedding | 기동 거부 |
| sparse 계약 | vocab.json meta(k1·b·avgdl·vocab_size) == manifest.sparse | 거부 |
| 토크나이저 | kiwipiepy major.minor == manifest (패치 차이 경고) | 거부 |
| 컬렉션 | 존재 + count == manifest.points(10,063) | 거부 |
| 용어 사전 | glossary.jsonl 로드(664건 로그) | 거부 |
| 임베딩 프로브 | "수행의무" 인코딩 L2노름 1±1e-3 | 거부 |

질의측 CPU-fp32 허용(계약 = 모델·리비전·normalize). 거부 메시지는 항목·기대값·실측값·조치 힌트 포함. `routing_gold.json`은 검증 대상도, 로드 대상도 아니다.

## 4. 공통 응답 규약

모든 도구 출력 최상위에 `"collection"` 포함(인용 판본 고정 — 규약 3.1). 오류 봉투:
```json
{"error": {"code": "NOT_FOUND | INVALID_INPUT | CONTRACT_MISMATCH | UPSTREAM_UNAVAILABLE",
           "message": "무엇이 실패했는가", "hint": "다음 행동"}}
```
텍스트 절단 금지. 총량은 상한으로만: top_k ≤ 20, context ≤ 3, terms ≤ 10, definitions ≤ 5.

## 5. 도구 3종 — 입출력 형식

### 5.1 `standards_get_paragraph` — 문단 직조회·검증 (질의 ④·⑤)

입력:
```json
{"cid": "KIFRS::1115::31",   // 필수. 복합 ID(논리). '#순번' 없이
 "context": 0}                // 선택 0~3. 같은 기준서 seq ±N 이웃
```
출력:
```json
{"collection": "standards_20250829_bgem3", "found": true,
 "paragraphs": [
   {"cid": "KIFRS::1115::31", "source_type": "회계기준", "standard_no": "1115",
    "standard_title": "고객과의 계약에서 생기는 수익", "para_no": "31",
    "para_type": "본문", "section_path": "수행의무의 이행", "seq": 74,
    "text": "...", "is_context": false}],
 "notes": []}
```
동작: UUID5 직조회 → 실패 시 payload 등가 조회(standard_no+para_no). 분할 문단은 전 조각을 part_no 순으로 하나의 text로 결합 + notes "분할 문단 N조각 재조립". context 이웃도 재조립 적용. 오류: cid에 `#` → `INVALID_INPUT` + hint "논리 ID로 조회 — '#' 앞부분 사용"; 미존재 → `NOT_FOUND` + hint "standards_search로 탐색".

### 5.2 `standards_search` — 하이브리드 검색 + 정책 + 정의 주입 (질의 ①·②·③·⑥)

입력:
```json
{"query": "수행의무 식별 절차",       // 필수 1~500자
 "standard_no": ["1115"],            // 선택. 조서에서 명시 참조를 발견했을 때의 라우팅 필터
 "source_type": ["회계기준"],         // 선택: 감사기준|회계기준|실무지침
 "para_type": null,                  // 선택: 예 "정의", "요구사항"
 "top_k": 8,                         // 기본 8, 1~20
 "include_examples": false}          // 예시류(부록) 스위치 — 문안·예시 작업 시 true
```
출력:
```json
{"collection": "...",
 "results": [
   {"cid": "KIFRS::1115::22", "score": 0.0164, "standard_no": "1115",
    "standard_title": "...", "para_no": "22", "para_type": "본문",
    "section_path": "수행의무의 식별", "text": "...", "notes": []}],
 "definitions": [
   {"term": "수행의무", "source_cid": "KIFRS::1115::정의-수행의무",
    "standard_no": "1115", "text": "..."}],
 "applied": {"filters": {"standard_no": ["1115"]},
   "fusion": "server-RRF (dense+sparse, prefetch 각 50)",
   "oov_query_tokens": [], "examples_excluded": 0, "notes": []}}
```
동작: dense(bge-m3)+sparse(kiwipiepy→vocab, 값 1.0; 어휘 밖 토큰은 무시·`oov_query_tokens` 기록) prefetch 각 50에 동일 필터, 서버 RRF. `include_examples=false`면 para_type=부록 제외 + `examples_excluded` 수와 재호출 힌트. para_type=참조 결과에는 per-result note "발췌 대조표 — 원전 문단 우선 인용"(3.7 v1 가드). 분할 히트는 재조립. 정의 주입: 질의 어휘 일치 최우선 → 결과 텍스트 빈도순, 캡 5, 동일 cid가 results에 있으면 생략, 충돌 시 문맥 기준서(필터 → 결과 최빈) 자체 정의 → 정의 보유 기준서, source_cid 필수. 이웃 확장은 이 도구 소관이 아님 — 도구 설명에 `standards_get_paragraph(context=N)` 안내.

### 5.3 `standards_define_terms` — 용어 정의 직조회

입력:
```json
{"terms": ["수행의무", "지배력"], "context_standard": "1115"}   // terms 1~10
```
출력:
```json
{"collection": "...",
 "definitions": [
   {"term": "수행의무", "matched": "수행의무",
    "source_cid": "KIFRS::1115::정의-수행의무", "standard_no": "1115",
    "source_type": "회계기준", "text": "...",
    "alternates": [{"source_cid": "KSA::1200::정의-…", "standard_no": "1200"}]}],
 "not_found": []}
```
매칭: 원문 정확 일치 → 공백 제거(term_norm) 일치. 복수 정의는 대표 1건 본문(context_standard 우선 → 정의 보유 기준서) + 나머지 alternates 포인터.

## 6. 에이전트 해석 플로우 — CLAUDE.md 탑재 지침 (신설)

아래 블록을 저장소 `CLAUDE.md`에 "조서 해석 플로우" 절로 추가한다. 이것이 콜드 해석의 방법론이며, **조서번호 체계나 조서별 근거 기준 같은 사전 지식을 담지 않는다** — 답은 문서와 도구에서 나와야 한다.

```markdown
## 조서 해석 플로우 (사용자가 조서 파일을 주며 해석을 요청할 때)

1. 읽기 — 파일 전체를 직접 연다(전 시트/전문). 제목·머리글·열 구성·점검 항목 문장·
   서명란·기준서 언급을 관찰하고, 관찰과 추론을 구분해 기록한다.
2. 정체 가설 — 관찰만으로 "무슨 작업을 위한 조서인가"의 가설을 세운다.
3. 근거 탐색 (MCP) —
   - 문서에 기준서·지침 번호가 명시돼 있으면: standards_search에 standard_no
     필터로 해당 기준의 요구사항을 확인한다 (문서가 준 단서는 콜드 해석의 일부다).
   - 명시 참조가 없으면: 점검 항목 문장을 질의로 자유 검색한다 (필터 없이 시작,
     결과가 특정 기준서로 수렴하면 그때 필터를 걸어 심화).
   - 낯선 용어는 standards_define_terms로 정의를 확보한다.
   - 이 조서가 문안·예시 작성용으로 판단되면 include_examples=true를 사용한다.
4. 산출 — reports/해석_{파일명}.md 로 저장:
   ## ① 이 조서의 정체 (2~4문장: 무엇을, 어느 감사 국면에서, 왜)
   ## ② 할 일 목록 (조서 항목 순서대로; 각 줄 = 수행 절차 + 근거 [cid])
   ## ③ 근거 규정 발췌 (②에서 인용한 문단의 원문 일부, cid별)
   ## ④ 미확인·불확실 사항 (근거를 못 찾은 항목, 가설로 남은 판단 — 솔직하게)
5. 인용 규율 — ②③의 모든 규정 주장에는 cid를 단다. 제출 전 인용 cid 중
   최소 3건을 standards_get_paragraph로 재조회해 본문과 대조한다.
   근거 없는 규정 주장은 쓰지 않는다 — 모르면 ④로 보낸다.
```

## 7. 평가 절차 (신설) — 골드셋 recall

1. **골드셋**: `eval/routing_gold.json` — 서식 46종의 명시 참조 스캔에서 파생(102 조서 항목). **한계를 명시한다**: 명시 참조 기반이므로 에이전트의 정당한 초과 인용(암묵 관련 기준)이 gold에 없을 수 있다 → **recall 중심** 지표, 초과분은 자동 감점하지 않고 사람 검토로 정당성 판단.
2. **채점기 `eval/score_interpretation.py`** — 입력: 해석 보고서 md 경로 + 조서 id / 처리: 보고서에서 cid 패턴(`[A-Z-]+::[^:\s]+::…`)을 추출해 standard_no 집합으로 환원, gold와 대조 / 출력:
```json
{"worksheet": "2100", "gold": ["KSA:315"], "cited": ["KSA:315", "KSA:240"],
 "recall": 1.0, "extras": ["KSA:240"], "extras_note": "사람 검토 대상(감점 아님)"}
```
3. v1.1 범위 = 파일럿 4종의 recall 기록. 46종 전량 캠페인은 다음 단계(구 4단계의 재정의)로 예고만.
